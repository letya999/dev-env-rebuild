# Contributing

## Как сообщить о проблеме

Если скрипт ведёт себя не так как ожидалось:

1. Открой [Issues](../../issues) → "New issue"
2. Укажи:
   - Версию Windows (`winver`)
   - Какой скрипт запускал и с какими флагами
   - Вывод скрипта (dry-run сначала, потом с `-Execute` если проблема в нём)
   - Что ожидал и что получил

## Как предложить изменение

1. Сделай fork репозитория
2. Создай ветку: `git checkout -b fix/название-проблемы`
3. Внеси изменения
4. Проверь dry-run перед тем как предлагать pull request
5. Открой Pull Request с описанием что и зачем изменил

## Правила для скриптов

- Каждый скрипт должен поддерживать флаг `-Execute` для переключения dry-run / реального режима
- В dry-run режиме скрипт только пишет что сделал бы, ничего не трогает
- Не хардкодить пути с именем пользователя — использовать `$env:USERPROFILE`
- Для npm-путей использовать `Get-NpmPrefix` из `01_uninstall/lib.ps1`
- Проверять существование объекта перед удалением

## Что не надо делать

- Не добавлять скрипты которые удаляют без `-Execute` флага
- Не хардкодить winget ID без проверки что пакет существует
- Не игнорировать ошибки молча — пиши в вывод что пошло не так
