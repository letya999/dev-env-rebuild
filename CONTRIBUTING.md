# Contributing

## Как сообщить о проблеме

Если скрипт ведёт себя не так как ожидалось:

1. Открой [Issues](../../issues) → "New issue"
2. Укажи:
   - ОС и версию (`winver`, `sw_vers` или `/etc/os-release`)
   - Какой скрипт запускал и с какими флагами
   - Вывод скрипта (dry-run сначала, потом с `-Execute`/`--execute` если проблема в нём)
   - Что ожидал и что получил

## Как предложить изменение

1. Сделай fork репозитория
2. Создай ветку: `git checkout -b fix/название-проблемы`
3. Внеси изменения
4. Проверь dry-run перед тем как предлагать pull request
5. Открой Pull Request с описанием что и зачем изменил

## Правила для скриптов

- Каждый destructive-скрипт должен поддерживать dry-run по умолчанию
- Windows-скрипты используют флаг `-Execute` для реального режима
- macOS/Ubuntu-скрипты используют флаг `--execute` для реального режима
- В dry-run режиме скрипт только пишет что сделал бы, ничего не трогает
- Не хардкодить пути с именем пользователя — использовать `$env:USERPROFILE` в PowerShell и `$HOME` в Bash
- Для npm-путей использовать `npm config get prefix` или существующий helper выбранной ОС
- Проверять существование объекта перед удалением
- Не удалять project roots, `.git`, существующие SSH-ключи, `~/.claude`, `~/.codex`, `~/.gemini` без отдельного ручного backup

## Что не надо делать

- Не добавлять скрипты которые удаляют без явного execute-флага
- Не хардкодить winget ID без проверки что пакет существует
- Не игнорировать ошибки молча — пиши в вывод что пошло не так
